[[section-runtime-view]]
== Runtime View


ifdef::arc42help[]
[role="arc42help"]
****
.Contents
The runtime view describes concrete behavior and interactions of the system’s building blocks in form of scenarios from the following areas:

* important use cases or features: how do building blocks execute them?
* interactions at critical external interfaces: how do building blocks cooperate with users and neighboring systems?
* operation and administration: launch, start-up, stop
* error and exception scenarios

Remark: The main criterion for the choice of possible scenarios (sequences, workflows) is their *architectural relevance*. It is *not* important to describe a large number of scenarios. You should rather document a representative selection.

.Motivation
You should understand how (instances of) building blocks of your system perform their job and communicate at runtime.
You will mainly capture scenarios in your documentation to communicate your architecture to stakeholders that are less willing or able to read and understand the static models (building block view, deployment view).

.Form
There are many notations for describing scenarios, e.g.

* numbered list of steps (in natural language)
* activity diagrams or flow charts
* sequence diagrams
* BPMN or EPCs (event process chains)
* state machines
* ...


.Further Information

See https://docs.arc42.org/section-6/[Runtime View] in the arc42 documentation.

****
endif::arc42help[]

This section displays sequence diagrams. These show the interaction of objects in different scenarios.

=== Game lifecycle against the computer (PvE)

This diagram illustrates the standard single-player workflow. The React frontend communicates with the Node.js backend, which internally delegates the heavy AI computation to the Rust Game Engine.

[plantuml]
----
@startuml
actor User
participant "React SPA" as WebApp
participant "API Gateway" as Gateway
participant "Users API\n(Node.js)" as UsersService
participant "GameY Engine\n(Rust)" as Gamey
database MongoDB as Db

User -> WebApp: Select PvE mode & difficulty
WebApp -> Gateway: POST /api/matches/new
Gateway -> UsersService: Forward
UsersService -> Db: Create match record
UsersService --> WebApp: Match created (Initial YEN)

loop Game loop (Player Turn)
    User -> WebApp: Makes move on Hex board
    WebApp -> Gateway: POST /api/bot/play (User YEN)
    Gateway -> UsersService: Forward
    UsersService -> Gamey: HTTP POST /api/gamey/calculate
    Gamey -> Gamey: Process rules & generate Bot move
    Gamey --> UsersService: Return new state (Bot YEN)
    UsersService -> Db: Update match state
    UsersService --> Gateway: 200 OK (New YEN state)
    Gateway --> WebApp: Display bot move
    WebApp --> User: Render updated board

    alt Game ends (Win/Lose/Draw)
        UsersService -> Db: Mark match as finished & update stats
    end
end
@enduml
----

=== Bot against Application

This diagram shows the life cycle of the game in which a bot participates against the machine.

[plantuml]
----
@startuml
actor Bot
participant API
participant Gamey
Bot -> API: Select strategy and game type
API -> Gamey: Initialize game
Gamey --> API: Initial board
loop Game loop
    Bot -> API: Send move (YEN)
    API -> Gamey: Apply move
    Gamey -> Gamey: Check game status
    alt Game ended
        Gamey --> API: Result
        API --> Bot: Return result
        break
    else Game continues
        API -> Gamey: Request machine move
        Gamey --> API: Returns movement (YEN)
        API --> Bot: Return move
    end
end
@enduml
----

=== User registration

This diagram shows the different interactions that take place when a user registers, routing through the API Gateway and securing the password.

[plantuml]
----
@startuml
actor User
participant "React SPA" as WebApp
participant "API Gateway\n(Nginx)" as Gateway
participant "Users API\n(Node.js)" as UsersService
database MongoDB as Db

User -> WebApp: Submit registration form
WebApp -> Gateway: POST /api/users/register
Gateway -> UsersService: Forward request
UsersService -> Db: Check if user exists
alt User already exists
    Db --> UsersService: User found
    UsersService --> Gateway: 409 Conflict (User exists)
    Gateway --> WebApp: 409 Conflict
    WebApp --> User: Show error message
else New user
    Db --> UsersService: User not found
    UsersService -> UsersService: Hash password (e.g., Bcrypt)
    UsersService -> Db: Save new user document
    Db --> UsersService: Confirmation
    UsersService --> Gateway: 201 Created
    Gateway --> WebApp: 201 Created
    WebApp --> User: Show confirmation & redirect to Login
end
@enduml
----

=== Real-time Multiplayer Match (PvP)

This diagram details the WebSocket communication used for real-time multiplayer games, handled by the dedicated Multiplayer service.

[plantuml]
----
@startuml
actor "Player 1" as P1
actor "Player 2" as P2
participant "API Gateway" as Gateway
participant "Multiplayer Service\n(Socket.io)" as Multi
database MongoDB as Db

P1 -> Gateway: Connect (WSS)
Gateway -> Multi: Upgrade to WebSocket
P2 -> Gateway: Connect (WSS)
Gateway -> Multi: Upgrade to WebSocket

P1 -> Multi: Emit "join_queue"
P2 -> Multi: Emit "join_queue"
Multi -> Multi: Match P1 and P2 (Create Room)
Multi -> Db: Create multiplayer match record
Multi --> P1: Emit "match_started" (Initial YEN)
Multi --> P2: Emit "match_started" (Initial YEN)

opt Gameplay Loop
    P1 -> Multi: Emit "player_move" (YEN)
    Multi -> Multi: Validate move logic
    Multi -> Db: Save state
    Multi --> P2: Emit "opponent_move" (YEN)
    
    P2 -> Multi: Emit "player_move" (YEN)
    Multi -> Db: Save state
    Multi --> P1: Emit "opponent_move" (YEN)
end

P1 -> Multi: Emit "surrender"
Multi -> Db: Update match stats (P2 Wins)
Multi --> P2: Emit "opponent_surrendered"
@enduml
